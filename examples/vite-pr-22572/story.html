    Stop Windows from serving your .env through alternate file-name spellings

    On Windows, one file answers to several names — and the dev server's deny list only knew the official one. The fix refuses the two characters every alias depends on.

    Act 1 · The picture

    Vite's dev server keeps a “never serve these” list — .env, keys, certificates — so a browser can't fetch your secrets even from an allowed folder. On Windows, that guard could be walked around, because on Windows the same file answers to several names: an old DOS-style nickname (ENV~1) and a “stream” suffix spelling (.env::$DATA). The deny list checked only the official name. Ask by nickname, and Windows happily handed the file over. It only bites when the dev server is exposed to the network (--host) — which is exactly when it matters.

    The fix doesn't teach the deny list every Windows spelling. It does something blunter: every alias needs one of two characters — ~ or : — so any path carrying either is refused before any other check runs.

    Act 2 · The journey

    The request arrivesTwo characters refused on sight

    Every file the dev server serves passes through one gate. The PR adds the two refusals at the very top of that gate — before the deny list and even before the allow list — so nothing downstream can rescue a path the guards reject. The tilde guard runs only on Windows; the colon guard runs everywhere, with one excused colon: a real Windows path legitimately starts with a drive letter (C:/…), so that prefix is stripped before checking.

    Plain words · Windows alternate paths

    Like a person with a legal name, a nickname, and an old maiden name: all three reach the same human, but a guest list that checks only the legal name lets the other two walk in. Windows files have a compatibility nickname (ENV~1 — always containing ~) and a stream spelling (.env::$DATA — always containing :).

    Show me the code — the two guards and the drive-letter excuse

    windowsDriveRE is /^[A-Z]:/i (static.ts:289) — when it matches, the first two characters are dropped before the colon check, so C:/project lives and .env::$DATA dies.

    Proving itAsking Windows for a real attack name

    You can't hardcode the nickname in a test: Windows invents it per machine (ENV~1 here, ENV~2 there). So the test suite now asks Windows directly — a small PowerShell helper fetches the genuine short name of the playground's .env, the config injects it into the test page, and two new attack rows join the existing serve-matrix: the stream spelling and the live nickname. Where Windows won't produce a nickname, the test marks itself skipped instead of failing — that's the new skip flag, and the big indentation churn in the test file is just the body moving into the new call signature, not logic.
